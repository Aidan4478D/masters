\section{Methodology}

\subsection{YOLO-Based Object Detection}

Object detection is a computer vision task that involves both localizing objects
within an image and assigning them class labels. Unlike image classification, 
which predicts a single label for an entire image, object detection must determine
both what objects are present and where they are located.

In this work, vehicle detection is performed using the YOLO family of models.
YOLO processes an image in a single pass and directly outputs predicted bounding boxes, 
confidence scores, and class labels for the objects it detects.
This formulation allows fast inference while maintaining strong detection
performance, making YOLO well suited for traffic-camera imagery and other 
real-time scene-understanding tasks.

\subsubsection{Convolutional Neural Networks (CNNs)}
\label{sec:cnn}

Object detection models such as YOLO use convolutional neural networks (CNNs) to 
extract visual information from images \cite{redmon2016YOLO}. Unlike traditional 
neural networks that treat an image as a one-dimensional vector of pixel values, 
CNNs operate directly on the two-dimensional structure of an image. This allows 
the network to learn patterns based on relationships between neighboring pixels, 
such as edges, textures, and shapes \cite{geeks4geeks_2017}.

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=6.5cm]{figures/methodology/4_1_1_cnn_diagram.png}
    \caption{Example of convolutional neural network operations. A
             convolutional kernel slides across the input image and performs
             a dot product with local pixel values to produce a feature map
             highlighting the presence of specific patterns. In typical CNN
             architectures, a nonlinear activation function such as ReLU is
             applied after convolution. A pooling layer may then downsample
             the feature map while preserving the strongest activations.
             Figure adapted from \cite{islam_pooling_2025}.}
    \label{fig:cnn-diagram}
\end{figure}

CNNs analyze images using small filters, also called kernels, that slide across 
the image. At each position, the kernel performs a dot product with the underlying 
pixels to produce a single output value. Repeating this process across the image 
produces a feature map, which highlights where the pattern represented by the kernel 
appears in the image. During training, the network learns kernels that respond to 
useful visual patterns \cite{Goodfellow-et-al-2016, oshea2015introCNN}.

Several parameters control how these operations are applied. One important parameter
is the stride, which determines how far the kernel moves between successive positions.
When the stride is greater than one, the resulting feature map has a lower spatial
resolution than the original image. CNN architectures may also include pooling 
operations, which further reduce spatial resolution while preserving the strongest
responses. As convolution and pooling layers are applied repeatedly, each value
in a deeper feature map reflects information from a larger portion of the input 
image. This region is called the receptive field.
\cite{oshea2015introCNN}.

In Fig.~\ref{fig:cnn-diagram}, a simple binary image is processed using a 
$3 \times 3$ kernel with stride $1$, producing a feature map. This feature map 
is then max-pooled using a $2 \times 2$ filter with stride $2$, resulting in a 
$2 \times 2$ pooled feature map containing the maximum value from each region. 
As convolution and pooling operations are repeated throughout the network, feature 
maps become smaller while their receptive fields grow larger, meaning each location
summarizes information from a larger portion of the original image.

Because each feature map location corresponds to a region of the input image, CNNs 
preserve a spatial relationship between feature map positions and image regions. 
This property allows models to generate localized predictions across the image 
rather than producing a single output for the entire input. Object detection 
models such as YOLO leverage this spatial structure to generate predictions at 
multiple positions in the feature map, enabling the detection of objects in different
parts of the scene.

\subsubsection{Grid-Based Prediction}
\label{sec:grid-preds}

To perform object detection, the input image is conceptually divided into a $W \times H$
grid of cells, as illustrated in Figure~\ref{fig:yolo-grid}. In practice, this 
grid corresponds to the spatial resolution of the feature map produced by the 
convolutional neural network. As stated in the Section~\ref{sec:cnn}, because convolutional
layers often use strides greater than one and pooling operations reduce spatial 
resolution, the feature map is smaller than the original image.

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=6cm]{figures/methodology/4_1_2_grid.png}
    \caption{Conceptual grid created by the YOLO architecture. 
             Adapted from \cite{redmon2016YOLO}.}
    \label{fig:yolo-grid}
\end{figure}

\FloatBarrier

If the network has a stride of $s$, the number of grid cells along the horizontal 
and vertical axes is approximately:

\begin{equation}
    W_{grid} = \frac{W_{input}}{s}, \qquad H_{grid} = \frac{H_{input}}{s}
\end{equation}

Where $W_{input}$ and $H_{input}$ are the width and height of the input image 
respectively. Each grid location therefore corresponds to a region of the original 
image and serves as a prediction point. YOLO uses these spatial locations to 
produce candidate bounding boxes, objectness scores, and class probabilities for
objects whose centers fall within the corresponding image region. This grid-based
formulation enables the model to perform dense prediction across the image, 
allowing multiple objects to be detected simultaneously within a single
forward pass of the network \cite{redmon2016YOLO}.

% ------------------------------------------------------------------------------

\subsubsection{Bounding Box Prediction}

For each grid cell, the CNN predicts $B$ candidate bounding boxes along with
confidence scores and class probabilities. Each predicted bounding box is
parameterized by five values:

\begin{equation}
(x, y, w, h, c)
\end{equation}

where:
\begin{itemize}
    \item $x, y$ represent the center coordinates of the bounding box relative
    to the grid cell,
    \item $w, h$ represent the width and height of the bounding box relative
    to the image dimensions,
    \item $c$ represents the confidence score associated with the prediction.
\end{itemize}

These parameters jointly represent the location and size of the predicted object within the 
image. The bounding box predictions are conceptually shown in Figure~\ref{fig:yolo-bbox}.

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=6cm]{figures/methodology/4_1_3_bounding_box.png}
    \caption{Bounding boxes created by the YOLO architecture. The thickness of 
             each box corresponds to the model's confidence score. The higher the 
             confidence, the thicker the box. Adapted from \cite{redmon2016YOLO}.}
    \label{fig:yolo-bbox}
\end{figure}

The confidence score represents the model's belief that a predicted bounding box 
contains an object and is defined as:

\begin{equation}
c = P(\text{object}) \times \mathrm{IoU}^{\text{truth}}_{\text{pred}}
\end{equation}

Where $P(\text{object})$ is the probability that an object exists in the predicted region 
and $\mathrm{IoU}^{\text{truth}}_{\text{pred}}$ is the Intersection over Union 
(IoU) between the predicted bounding box and the ground truth bounding box \cite{redmon2016YOLO}. 
See Section~\ref{sec:iou} for more details regarding IoU. This formulation allows
the model to simultaneously capture both object presence and localization accuracy 
in a single prediction score.

% ------------------------------------------------------------------------------

\subsubsection{Class Probability Prediction}

In addition to predicting bounding boxes, each grid cell also produces a vector 
of conditional class probabilities representing the likelihood that the detected 
object belongs to each category. These probabilities are predicted independently
of the bounding box geometry and are conditioned on the presence of an object 
within the grid cell.

If the model is trained to detect $C$ object classes, each grid cell predicts:

\begin{equation}
P(\text{Class}_i \mid \text{object})
\end{equation}

for \(i = 1, 2, \ldots, C\). During inference, the final class-specific
confidence score for a predicted bounding box is computed by combining the
conditional class probability with the confidence score of that box \(c\). Then,

\begin{equation}
\text{Score}_i = c \times P(\text{Class}_i \mid \text{object})
\end{equation}

This gives a class-specific confidence score for each predicted bounding box, 
reflecting both the likelihood that an object is present and the probability that
the object belongs to a particular class. Conceptually, these class predictions 
are produced across the spatial grid of the image. Figure~\ref{fig:yolo-class-prob}
illustrates this idea, where the colors indicate the dominant predicted class at
different grid locations rather than the final post-processed detections \cite{redmon2016YOLO}.

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=6cm]{figures/methodology/4_1_4_class_prob.png}
    \caption{Conceptual illustration of class predictions across the YOLO grid. 
             Each colored region indicates the dominant predicted class at that spatial 
             location, based on the highest class-specific score. The figure is intended 
             to show how class predictions are distributed across the image and does not 
             represent the final post-processed detections. Adapted from \cite{redmon2016YOLO}.}

\end{figure}

\subsubsection{Post-Processing and Final Detections}

Because each grid cell may produce multiple candidate bounding boxes, the raw 
output of the network often contains several overlapping predictions that correspond
to the same object. To produce the final detections, a post-processing step known 
as Non-Maximum Suppression (NMS) is applied. Non-Maximum Suppression operates by:

\begin{enumerate}
    \item Ranking all predicted bounding boxes according to their confidence scores.
    \item Selecting the highest-scoring prediction.
    \item Removing any remaining predictions whose overlap with the selected box 
          exceeds a predefined IoU threshold.
    \item Repeating this process until only high-confidence predictions remain.
\end{enumerate}

This procedure removes redundant detections while preserving the most confident 
prediction for each object in the scene. The resulting set of bounding boxes are 
the final object detections, which include the predicted object class, bounding 
box coordinates, and associated confidence score, as shown in Figure~\ref{fig:yolo-predictions}.
For further information regarding the YOLO architecture and training procedure, 
please refer to the original work by Redmon et al. \cite{redmon2016YOLO}.

\subsubsection{YOLOv11 Architecture}
\label{sec:yolov11}

This study evaluated several YOLO variants, including YOLOv8, YOLOv11, and YOLOv26.
Because YOLOv11 was ultimately selected as the strongest-performing model in the later
experiments, its architecture is described here in more detail. Like the other YOLO
variants considered in this work, YOLOv11 follows the general detection pipeline
described in the previous sections, in which bounding boxes, object confidence scores,
and class probabilities are predicted directly from convolutional feature maps in a
single pass through the network. YOLOv11 is a recent member of the YOLO family that
builds on earlier YOLO detectors with architectural refinements aimed at improving
detection accuracy and computational efficiency.

Like many modern object detectors, YOLOv11 uses a backbone-neck-head architecture.
The backbone is a deep convolutional network that extracts visual features from the
input image. These features are then passed to a neck module, which combines
information across multiple spatial scales so that objects of different sizes can be
recognized more effectively. Finally, the detection head outputs bounding box
coordinates, object confidence scores, and class probability predictions at multiple
spatial locations in the feature maps \cite{YOLOv11paper}.

Compared with earlier YOLO models, YOLOv11 introduces several refinements intended to
improve both accuracy and efficiency. One important component is the C3k2 module, an
enhanced convolutional block designed to improve feature extraction while reducing
computational cost through more efficient reuse of intermediate feature
representations. YOLOv11 also uses PAN/FPN-style neck layers to improve feature
aggregation across multiple resolutions, which helps the detector handle objects at
different spatial scales. In addition, refinements to the detection head improve
bounding box localization and class prediction while preserving the fast inference
speed associated with YOLO-based detectors. Together, these changes make YOLOv11 a
strong and computationally efficient detector for real-time applications
\cite{YOLOv11paper}.

In this study, YOLOv11 was selected for the final detector because it achieved the
best empirical performance on the vehicle detection dataset while maintaining
inference efficiency suitable for large-scale traffic-camera imagery. Additional
details on the YOLOv11 architecture and training procedure are available in the
Ultralytics documentation \cite{yolo11_ultralytics} and in the work of Khanam and
Hussain \cite{YOLOv11paper}.

\subsection{MOVES-based Taxonomy}
\label{sec:moves-taxonomy}

For the experiments reported in this chapter, model training and evaluation were 
conducted using a final 6-class taxonomy: 
\verb|passenger_car|, \verb|passenger_truck|, \verb|light_commercial_truck|, 
\verb|bus|, \verb|truck|, and \verb|motorcycle|. 
This label set was selected to provide a practical balance between compatibility
with downstream MOVES-based emissions analysis and what could be distinguished 
reliably from low-resolution traffic-camera imagery. Rather than reproducing the
full MOVES source-type framework, the taxonomy preserves broad categories relevant 
to emissions modeling while reducing visual distinctions that are difficult to 
learn consistently.

To achieve this balance, fine-grained vehicle labels were grouped accordingly. 
Sedan instances were mapped to \verb|passenger_car|, while \verb|SUV| instances, 
\verb|pickup_truck| instances, and \verb|van| instances with the attribute \verb|is_minivan| 
instances were grouped into \verb|passenger_truck|. The EPA notes that it is 
difficult to visually determine whether a vehicle should be treated as a 
passenger car or a light-duty truck, but classifies many SUVs and minivans as 
light-duty trucks rather than passenger cars \cite{USEPA_def_truck_2019}. 

The remaining van instances were mapped to \verb|light_commercial_truck|. 
This separation is useful from a computer-vision standpoint, as commercial vans 
tend to have more consistent shapes and usage patterns than SUVs and pickup trucks.
It also aligns with MOVES technical guidance, which treats passenger trucks and 
light commercial trucks as separate source types \cite{us_epa_moves3_2020}. 
Additionally, since the original \verb|two_wheeler| class only contained 
motorcycles, it was remapped directly to \verb|motorcycle|.

Finally, \verb|bus| and \verb|truck| were preserved as independent classes because they are 
operationally different, visually distinct, and treated as separate categories 
within MOVES. Within the bus category, however, school buses were not modeled 
as a separate detector class, even though MOVES distinguishes other buses, 
transit buses, and school buses as separate source types. Instead, school buses
were retained as an annotation attribute and grouped into the broader bus class. 
This is because the primary objective of this study is to identify the broad 
vehicle categories most relevant to downstream emissions-oriented analysis rather
than to infer every possible subtype within each category. As such, the bus 
attribute, \verb|is_school|, was preserved as useful annotation metadata and a potential
avenue for future refinement, but it was not treated as essential to the core 
detector taxonomy. The final mapping between the dataset taxonomy and the MOVES 
taxonomy is shown in Figure~\ref{fig:moves-mapping}.

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=7cm]{figures/methodology/4_2_class_taxonomy.png}
    \caption{Final MOVES-based class mapping used in this study. Dataset classes 
         on the left were mapped to the MOVES-aligned detector classes on the 
         right. Arrows of the same color indicate the corresponding class 
         mapping, with van instances labeled \texttt{is\_minivan} grouped into 
         \texttt{passenger\_truck} and the remaining van instances mapped to 
         \texttt{light\_commercial\_truck}.}
    \label{fig:moves-mapping}
\end{figure}

This MOVES-based taxonomy was used for the remainder of the study as the common 
label mapping for the dataset and for all subsequent training, testing, and evaluation.

\subsection{Handling Unknown Vehicles}
\label{sec:unknown-vehicles}

As described in Section~\ref{sec:unk-vehicle-class}, some vehicles were 
annotated as \verb|unknown_vehicle| when they could be recognized as vehicles but 
could not be assigned confidently to one of the six supervised classes. In the 
detector pipeline, these regions were not treated as a seventh learnable class. 
Instead, they were handled as ignore regions during both dataset preparation and 
evaluation.

When making the dataloader for the model training splits, regions that were labeled
\verb|unknown_vehicle| during dataset annotation were masked in the images using 
a neutral fill value of \verb|RGB(114,114,114)|. This meant that the detector was trained 
only on the MOVES-based target classes, while ambiguous vehicles were masked so 
they would not be forced into an incorrect label or treated as background even 
though they were clearly vehicles. Because this masking occurred during dataset 
generation, it was applied before any augmentation pipeline. See Figure~\ref{fig:masking-vehicle-example} 
for an example of a training image before and after masking.

During validation and later test-time evaluation, the images were not modified. 
Instead, a custom ignore-aware evaluation procedure was used. The \verb|unknown_vehicle| 
boxes were reconstructed directly from the original CVAT XML annotations, and any
prediction overlapping one of these regions above an IoU threshold of 0.40 was 
removed before scoring. The remaining predictions were then matched against the 
MOVES-based classes to compute AP50 and mAP. This ensured that detections falling
on visually ambiguous vehicles were not counted as false positives, while preserving
the standard inference settings for confidence threshold, NMS IoU threshold, and 
maximum detections. This same ignore-region procedure was applied consistently 
across model architectures during training, validation, and test evaluation.

\subsection{Performance Metrics}

To determine if using computer vision is an adequate method to estimate 
emissions coefficients in New York City, the vehicle detection pipeline must be 
rigorously evaluated. Unlike standard image classification, object detection 
requires assessing two distinct tasks simultaneously: accurately localizing a 
vehicle within a complex urban environment and correctly classifying its 
emissions profile category (e.g., distinguishing a truck from a bus). 
This subsection details the metrics selected to quantify 
the model's performance. It begins by establishing the spatial evaluation 
criteria, builds toward aggregate threshold-agnostic metrics like mean average 
arecision (mAP), and concludes by justifying these choices against traditional 
classification metrics that fall short in dynamic spatial contexts.

\subsubsection{Intersection over Union}
\label{sec:iou}

Before assessing the classification accuracy of the model, it is necessary to
quantify the spatial accuracy of the predicted bounding boxes. This is achieved
using the Intersection over Union (IoU) metric. IoU measures the overlap between
a predicted bounding box and a ground-truth bounding box. It is calculated as
the area of their intersection divided by the area of their union. Figure~\ref{fig:intersection-union}
illustrates the intersection and union regions for an example prediction, while
Figure~\ref{fig:iou_calc} shows the corresponding IoU calculation.

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=3.5cm]{figures/methodology/4_4_intersection_union.png}
    \caption{Example of the intersection and union between a ground-truth bounding box ({\color{red}red}) 
             and a predicted bounding box ({\color{myyellow}gold}). The figures show the original overlap, 
             the intersection area, and the union area used in the IoU calculation.}
    \label{fig:intersection-union}
\end{figure}

\begin{figure}[!htbp]
    \centering
    \includegraphics[width=0.6\textwidth]{figures/methodology/4_4_iou_calculation.png}
    \caption{IoU Calculation}
    \label{fig:iou_calc}
\end{figure}

An IoU score ranges from 0 (no overlap) to 1 (perfect alignment). By
establishing an IoU threshold (ex., 0.50), we define the strictness required 
for a bounding box prediction to be considered valid.

\subsubsection{Confusion Matrix}

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=6.5cm]{figures/methodology/4_4_confusion_matrix.png}
    \caption{Confusion Matrix}
    \label{fig:conf_matrix}
\end{figure}

With an IoU threshold in place, the standard confusion-matrix categories shown in
Figure~\ref{fig:conf_matrix} can also be applied to object detection. Unlike image
classification, where each image is assigned a single label, object detection must
evaluate both what the object is and where it is located. Because predicted bounding
boxes will rarely match the annotated box exactly, the IoU threshold provides a
practical way to decide whether a detection is close enough to count as correct. In
this way, the IoU threshold converts box overlap into the discrete categories needed
for evaluation, such as true positives, false positives, and false negatives. Within
the context of this project, the entries of the confusion matrix correspond to:

\begin{itemize}
    \item \textbf{True Positive (TP)}: A detection that matches a human-annotated
    ground truth box of the correct vehicle class with an IoU $\geq$ the defined 
    threshold. For example, the model successfully draws a bounding box around a 
    sedan and correctly classifies it as a Light-Duty Vehicle (LDV) with an IoU 
    that exceeds the threshold.

    \item \textbf{False Positive (FP)}: A predicted box that does not satisfy the
    matching criteria for a correct detection. In this study, false positives
    generally occur in three ways:
    \begin{enumerate}
        \item \textit{Misclassification}: The predicted box overlaps a ground-truth
        object sufficiently, but the assigned class label is incorrect.

        \item \textit{Poor Localization}: The predicted class is correct, but the
        bounding box does not overlap the ground-truth box enough to satisfy the
        required IoU threshold.

        \item \textit{Background/Duplicate}: The predicted box either does not
        correspond to any ground-truth object or duplicates another prediction for
        the same object.
    \end{enumerate}

    \item \textbf{False Negative (FN)}: A ground-truth vehicle that is not successfully
    matched by any prediction. This can occur when the model fails to detect the
    vehicle entirely, assigns it the wrong class, or predicts a bounding box that
    does not overlap sufficiently to meet the IoU threshold.

    \item \textbf{True Negative (TN)}: Conceptually, a true negative represents
    every possible empty region in an image (such as patches of bare asphalt, 
    sidewalks, buildings, or sky) where the model correctly predicted that \textit{no} 
    vehicle was present. Because the number of empty background regions in a 
    dense urban scene is effectively infinite, True Negatives mathematically 
    distort traditional evaluation metrics and are therefore excluded from 
    object detection performance calculations.
\end{itemize}

\subsubsection{Precision-Recall Curve}
Using the \textit{TP}, \textit{FP}, and \textit{FN} counts defined above, we can calculate 
precision and recall to evaluate the pipeline's ability and reliability in tracking 
urban traffic.

\begin{itemize}
    \item \textbf{Precision} measures the reliability of the model’s vehicle predictions.

    \begin{equation}
    Precision = \frac{TP}{TP + FP}
    \end{equation}

    In the context of this project, out of all the bounding boxes the model predicted 
    for a certain class (e.g. claiming to detect 50 trucks), what fraction were correct? 
    A high precision score ensures the model is not predicting false ``ghost'' 
    vehicles or misclassifying vehicle types.

    \item \textbf{Recall} measures the model's ability to find vehicles in the frame.

    \begin{equation}
    Recall = \frac{TP}{TP + FN}
    \end{equation}

    Out of all the actual ground truth objects present, what 
    fraction did the model successfully detect? A high recall score ensures 
    emission sources are not undercounted in the final analysis.
\end{itemize}

Modern object detectors like the YOLO output a confidence score with each 
predicted bounding box, creating a trade-off between precision and 
recall. Lowering the confidence threshold allows the model to predict more 
vehicles (increasing recall), but it simultaneously increases the risk of 
capturing background noise as valid vehicles (false positives), thereby 
lowering precision. By calculating precision and recall at every possible 
confidence threshold and plotting the results, we generate the precision-recall 
(PR) curve.

% \begin{figure}[!htbp]
%     \centering
%     \includegraphics[height=6.5cm]{figures/methodology/4_4_example_pr_curve.png}
%     \caption{Example Precision-Recall Curve}
%     \label{fig:ex_pr_curve}
% \end{figure}

The PR curve illustrates the model's overarching stability and allows us to 
visualize the exact penalty in precision when we tune the model to capture a 
higher volume of traffic.

\subsubsection{Average Precision and mAP}

To quantify the trade-off depicted in the PR curve into a single scalar value, 
we use the average precision (AP). AP represents the total area under the PR 
curve, depicted in Figure~\ref{fig:ex_pr_curve_ap}. It represents the model's 
reliability across every possible confidence threshold for a single specific 
vehicle class (e.g., evaluating solely the model's ability to isolate trucks
from the rest of the traffic flow). AP is calculated as follows:

\begin{figure}[!htbp]
    \centering
    \includegraphics[height=6cm]{figures/methodology/4_4_example_pr_curve_ap.png}
    \caption{Average Precision within a Precision-Recall Curve}
    \label{fig:ex_pr_curve_ap}
\end{figure}


\begin{equation}
AP = \int_{0}^{1} p(r) dr
\end{equation}

where $p(r)$ is the precision at recall $r$. By calculating the area under the 
curve, AP penalizes models that suffer severe drops in precision when attempting 
to retrieve the hardest-to-find vehicles (such as heavily occluded trucks).

To evaluate the air quality modeling pipeline across the entire vehicle, the 
individual AP scores for all distinct classes are averaged to produce the 
mean average precision (mAP):

\begin{equation}
mAP = \frac{1}{N} \sum_{i=1}^{N} AP_i
\end{equation}

where $N$ is the total number of classes ($N=5$ in this study). 
Because the mAP score relies on the initial spatial overlap requirement, standard 
evaluation protocols report performance at different Intersection over Union (IoU) 
thresholds. A common baseline is AP50 (AP calculated at an IoU threshold of 0.50), which 
tolerates moderate bounding box looseness. However, modern object detection systems
evaluate performance using the protocol established by the Microsoft Common 
Objects in Context (MS COCO) dataset \cite{lin_microsoft_2015}. The COCO-style mAP (often denoted 
as $AP@[.50:.05:.95]$) averages the mAP across ten distinct IoU thresholds ranging 
from 0.50 to 0.95 with a step size of 0.05:

\begin{equation}
mAP_{COCO} = \frac{1}{10} \sum_{t \in T}mAP_t
\end{equation}

Where $T = \{0.50, 0.55, 0.60, \dots, 0.95\}$ and $mAP_t$ is the mean average 
precision evaluated at IoU threshold $t$. 

The primary objective of this thesis is accurate vehicle detection and classification
for emissions-oriented traffic analysis, which makes AP50 an intuitive metric because
it measures whether a vehicle was detected with moderate spatial overlap. However,
this study reports both AP50 and COCO-style mAP, and uses mAP as the primary
performance indicator for several reasons:
\begin{itemize}
    \item First, mAP evaluates detection performance across multiple IoU thresholds
    from 0.50 to 0.95 rather than at a single cutoff, providing a more complete
    measure of localization quality.

    \item Second, localization accuracy is especially important in low-resolution
    urban traffic imagery, where vehicles are often small, partially occluded, and
    closely spaced. In these conditions, a loose bounding box may still count as
    correct under AP50 while failing to isolate the vehicle precisely.

    \item Third, in congested scenes, detections that overlap multiple nearby
    vehicles can appear acceptable at IoU $= 0.50$ even when they are too coarse
    for reliable counting and class-specific analysis. Using mAP helps penalize
    these cases more appropriately.

    \item Finally, COCO-style mAP is the standard summary metric in modern object
    detection literature, making the results easier to compare with prior work.
\end{itemize}

COCO-style mAP therefore provides a more rigorous measure of detector quality in 
this study, since it rewards models that not only detect vehicles, but also localize
them precisely in dense urban traffic scenes.

\subsection{Experimental Environment}
All detector training and evaluation experiments were conducted in Python using 
Google Colab with an NVIDIA L4 GPU. YOLO models were implemented and trained using 
the Ultralytics library, while alternative detector models were implemented in 
PyTorch. Data augmentation was applied using the Albumentations library where 
relevant. Experiments were run with a fixed seed of 42 to improve consistency 
across runs.
